Deep Copy vs Shallow Copy
Memory Mutation safety
Critical

Shallow copy duplicates the outer container but keeps nested object references pointing to the same memory. Deep copy recursively clones every nested object вҖ” fully independent.

вҡЎ ICPA context: When cloning a LangGraph StateGraph state dict between agent checkpoints, a shallow copy causes agents to mutate each other's claim data. Deep copy is mandatory.

# Shallow vs Deep Copy in Python
import copy

# ICPA: claim state snapshot
claim_state = {
"case_type": "DST",
"entities": ["Schaden_Nr_123", "Datum"],
"metadata": {"confidence": 0.91}
}

# Shallow вҖ” nested dicts SHARED
shallow = copy.copy(claim_state)
shallow["metadata"]["confidence"] = 0.5
print(claim_state["metadata"]["confidence"]) # 0.5 вҶҗ BUG!

# Deep вҖ” fully independent
deep = copy.deepcopy(claim_state)
deep["metadata"]["confidence"] = 0.5
print(claim_state["metadata"]["confidence"]) # 0.91 вң“ safe

# When to use each:
# copy.copy() вҶ’ flat structs (no nesting), performance critical
# copy.deepcopy() вҶ’ state machines, LangGraph, config objects
// Go equivalent вҖ” manual deep copy via JSON marshal trick

package main

import (
"encoding/json"
"fmt"
)

type Metadata struct {
Confidence float64 `json:"confidence"`
}

type ClaimState struct {
CaseType string `json:"case_type"`
Entities []string `json:"entities"`
Meta Metadata `json:"metadata"`
}

// Shallow: struct assignment (nested slices SHARED)
func shallowCopy(s ClaimState) ClaimState {
return s // struct fields copied, slice header SHARED
}

// Deep: JSON round-trip (simple, safe)
func deepCopy(src ClaimState) (ClaimState, error) {
var dst ClaimState
b, err := json.Marshal(src)
if err != nil { return dst, err }
return dst, json.Unmarshal(b, &dst)
}

// to avoid reflection overhead of JSON

Python Generators
Lazy eval Memory efficient
Core

Generators use yield to produce values one at a time, suspending execution and resuming on the next call. No intermediate list is ever built in memory.

вҡЎ ICPA context: Streaming LLM token output from Cohere/Bedrock, processing large German email batches from S3, and ChromaDB result iteration all use generators to avoid OOM on EKS pods.

# Generator for streaming ICPA email chunks
from typing import Generator, Iterator
import boto3

def stream_emails_from_s3(
bucket: str, prefix: str
) -> Generator[dict, None, None]:
s3 = boto3.client("s3")
paginator = s3.get_paginator("list_objects_v2")
for page in paginator.paginate(Bucket=bucket, Prefix=prefix):
for obj in page.get("Contents", []):
body = s3.get_object(
Bucket=bucket, Key=obj["Key"]
)["Body"].read().decode("utf-8")
yield {"key": obj["Key"], "content": body}

вң“ Generators Win

  • O(1) memory regardless of dataset
  • Pipeline composition (gen of gens)
  • Natural streaming for LLMs
  • Early exit on first match

вң— Generators Lose

  • Can only iterate once
  • No random access / len()
  • Harder to debug mid-stream
  • No parallelism (GIL)

Business impact: Replacing list comprehensions with generator expressions in ICPA's BGE-M3 embedding pipeline reduced peak EKS pod memory by ~60%, preventing OOM-kill restarts and improving p99 latency from 4.2s вҶ’ 1.8s per claim.

Generator vs List
Decision guide

The choice determines memory footprint at runtime. Lists materialise everything upfront; generators produce on demand.

Generator вҖ” use yield
  • Infinite or very large sequences
  • Stream processing / LLM tokens
  • Pipeline of transformations
  • Read-once data (S3, Kafka consumer)
  • Early exit patterns
List вҖ” use return
  • Need len(), random access, index
  • Multiple iterations required
  • Pass to functions expecting a list
  • Small dataset (<1M items)
  • Sorting or deduplication needed
# Memory comparison вҖ” ICPA embedding batch
import sys

texts = [f"Versicherungsschaden {i}" for i in range(100_000)]

# List: all in memory now
list_result = [t.upper() for t in texts]
print(sys.getsizeof(list_result)) # ~800KB+

# Generator: zero overhead until consumed
gen_result = (t.upper() for t in texts)
print(sys.getsizeof(gen_result)) # 112 bytes always

Async vs Reactive Code

Async: non-blocking I/O using event loop (Python asyncio, Go goroutines). Request fires, awaits response, frees thread. Reactive: data streams with backpressure вҖ” producer/consumer communicate via signals to prevent overflow.

// Go: goroutines + channels = native async+reactive

func ParallelAgents(ctx context.Context, text string) {
var wg sync.WaitGroup
results := make(chan AgentResult, 3)
errors := make(chan error, 3)

agents := []AgentFunc{
ClassifyAgent, ExtractAgent, TranslateAgent,
}
for _, agent := range agents {
wg.Add(1)
go func(a AgentFunc) {
defer wg.Done()
r, err := a(ctx, text)
if err != nil { errors <- err; return }
results <- r
}(agent)
}
// Close channels after all goroutines done
go func() { wg.Wait(); close(results) }()
}

if val: return json.loads(val)

lock_key = f"lock:{key}"

await r.delete(lock_key)
return result
# Another process is computing; brief wait

The choice is about data shape, consistency needs, and access patterns вҖ” not just scale.

SQL (Relational)
  • Fixed schema, ACID transactions
